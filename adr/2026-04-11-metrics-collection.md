# Архитектура и развитие системы сбора метрик

## Контекст

В проекте реализован самописный сбор метрик для отслеживания популярности функционала сайта (какие страницы и фичи востребованы) и мониторинга его работоспособности (аномальное падение количества событий может сигнализировать об ошибках).

Текущая реализация состоит из следующих компонентов:
1. **Бэкенд:** Метрики представлены событиями в пространстве `App\Events\Stats\`. Класс `WildcardMetricsListener` перехватывает все события этого пространства и через `PushMetricAction` отправляет их в Redis Stream.
2. **Фронтенд:** Для избежания избыточных запросов (бомбардировки бэкенда), события складываются в локальный массив через класс `Beacon`. Отправка на бэкенд происходит партиями (batching) через `navigator.sendBeacon` при достижении порога (100 событий) или при закрытии/скрытии страницы (`visibilitychange` / `pagehide`). Контроллер `BeaconController` принимает массив событий, сопоставляет их с разрешенным списком (whitelist) и генерирует соответствующие `App\Events\Stats\*` события на бэкенде.
3. **Агрегация:** Консольная команда `app:metrics:process` вычитывает сырые события из Redis Stream, агрегирует их в оперативной памяти с помощью сервисов (например, `MetricsAggregator`, `ViewsAggregator` и т.д.) и сохраняет результаты (увеличивает значения счетчиков) в базу данных MySQL (`metrics`, `views`, `image_views` и др.).

**Проблема:**
Текущая архитектура основана на простых счетчиках. События не имеют дополнительных параметров (payload / labels). Из-за этого невозможно проводить глубокую аналитику. Например, в тренажерах фиксируется сам факт старта, но неизвестно, какие настройки и языки были выбраны пользователем. Внедрение параметров при текущем подходе требует либо создания десятков новых классов-событий, либо серьезной переработки схемы БД.

## Решение

Текущее решение с Redis Stream в качестве буфера и отправкой метрик с фронтенда пачками (batching) через `navigator.sendBeacon` признано **успешным и эффективным** для текущего объема нагрузки. Оно защищает MySQL от большого количества мелких запросов на запись и не замедляет обработку HTTP-запросов.

Для решения проблемы отсутствия параметров предлагается рассмотреть несколько путей развития системы (см. Alternatives).

## Альтернативы

Ниже представлены варианты эволюции системы сбора метрик, от простых (модификация текущего кода) до внедрения новых инструментов.

### Вариант 1. Модернизация текущего решения (Добавление JSON Labels)

Изменение схемы агрегации и таблицы в MySQL для поддержки параметров (ярлыков/тегов).
- **Реализация:**
  1. Добавить в таблицу `metrics` колонку `labels` (тип JSON) или создать таблицу-справочник для тегов.
  2. В классы событий `App\Events\Stats\*` добавить поддержку массива `$labels`.
  3. В `MetricsAggregator` изменить ключ агрегации с `$eventName` на хэш `$eventName + json_encode($labels)`.
  4. В БД делать `INSERT ... ON DUPLICATE KEY UPDATE` по уникальному индексу `(date, event, labels)`.
- **Плюсы:** Не требует внедрения новых сервисов и инфраструктуры. Все остается в рамках PHP и MySQL.
- **Минусы:** Агрегация и построение отчетов в MySQL по JSON-полям на больших объемах данных будет работать медленно. Требуется аккуратная нормализация ключей в JSON, чтобы избежать дублей ({"lang":"ja","mode":"hard"} и {"mode":"hard","lang":"ja"}).

### Вариант 2. Использование ClickHouse (Сырые события)

Отказ от предварительной агрегации счетчиков в PHP в пользу хранения сырых событий.
- **Реализация:** Данные из Redis Stream через консольную команду пачками (batch insert) пишутся напрямую в ClickHouse в виде лога событий (дата, время, название события, user_id, JSON-параметры).
- **Плюсы:** Идеальное решение для аналитики. ClickHouse легко переваривает миллиарды записей и позволяет строить любые срезы по параметрам на лету.
- **Минусы:** Требует развертывания и поддержки новой СУБД (ClickHouse).

### Вариант 3. Использование систем продуктовой аналитики (PostHog / Matomo / Plausible / Yandex.Metrica)

Передача ответственности за сбор и аналитику параметров в специализированный инструмент.
- **Реализация:** Фронтенд напрямую, либо бэкенд асинхронно, отправляет сырые события с параметрами в API системы продуктовой аналитики. Для сохранения контроля над данными можно использовать self-hosted решения (PostHog или Matomo).
- **Плюсы:** Множество готовых графиков, воронок, анализ когорт "из коробки". Не нужно писать и поддерживать свой код для агрегации и визуализации.
- **Минусы:** Зависимость от стороннего решения (либо необходимость администрировать тяжелое self-hosted приложение, если это PostHog).

### Вариант 4. Использование Prometheus (TSDB)

Переход на классическую модель метрик с лейблами.
- **Реализация:** Вместо сохранения счетчиков в MySQL, они публикуются (или собираются Prometheus-ом) в формате TSDB.
- **Плюсы:** Стандарт индустрии. Prometheus изначально спроектирован для счетчиков с лейблами: `trainer_started_total{language="ja", mode="hard"} 1`. Идеальная связка с Grafana.
- **Минусы:** Prometheus больше заточен под системные метрики серверов, а не продуктовую аналитику (хотя и может использоваться для нее). Требует поднятия инфраструктуры Prometheus + Grafana.

## Статус

Предложено.

## Последствия

- Текущий подход с `navigator.sendBeacon` и `Redis Stream` остается без изменений, так как полностью оправдал себя с точки зрения производительности и устойчивости к нагрузкам.
- Для поддержки параметров в метриках необходимо выбрать один из предложенных вариантов (см. Альтернативы). Если требуется минимальное вмешательство в инфраструктуру, приоритетным является **Вариант 1**. Если в будущем потребуется глубокий анализ поведения пользователей и воронок, рекомендуется рассмотреть **Вариант 3 (PostHog)** или **Вариант 2 (ClickHouse)**.
