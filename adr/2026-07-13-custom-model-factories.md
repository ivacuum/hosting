# Собственные фабрики моделей

## Контекст

Тестам и сидерам проекта нужны не отдельные строки в таблицах, а согласованные графы моделей. Например, для поездки обязательны город, страна и пользователь. Значения этих моделей связаны между собой и должны соответствовать правилам предметной области: даты поездки не идут в обратном порядке, заголовки могут наследоваться от города, полиморфные связи получают правильный тип, а связанные комментарии, теги и токены создаются после родительской модели.

В проекте для этого используются собственные фабрики из `app/Factory` и `app/Domain/*/Factory`. Это обычные PHP-билдеры с единообразным API:

```php
$trip = TripFactory::new()
    ->withUser()
    ->create();
```

Такой вызов создает поездку и по необходимости весь граф ее зависимостей: пользователя, город и страну. В существующую связь можно передать идентификатор или модель, а для настройки новой зависимости — другую фабрику:

```php
$trip = TripFactory::new()
    ->withUser(UserFactory::new()->withLogin('phpunit'))
    ->withCity($city)
    ->create();
```

Методы `with*` клонируют фабрику. Поэтому настроенные фабрики можно безопасно переиспользовать как шаблоны, не накапливая состояние между вызовами.

Удобство разработки (developer experience, DX) является самостоятельной целью API фабрик. Частые и значимые состояния должны выражаться именованными методами предметной области. Например, `ImageFactory::new()->obsolete()` быстрее прочитать и сложнее неверно интерпретировать, чем `ImageFactory::new()->withUpdatedAt(now()->subMonths(7))`. Такие методы не только сокращают вызов, но и документируют типичные состояния модели, делают их доступными через автодополнение IDE и скрывают несущественные для теста технические детали.

Важное требование проекта — окончательно настроить каждую модель до ее сохранения. Каждая новая модель должна создаваться одним `INSERT`, без последующего `UPDATE` только ради применения состояния фабрики. Например, переданная в `withUser(UserFactory $factory)` фабрика пользователя должна получить нужные логин, почту, локаль и статус до первого и единственного сохранения пользователя.

Стандартные фабрики Laravel 13 применяют `state()` и атрибуты из `create()` до первого `save()`. Поэтому для обычных полей корневой модели они позволяют выполнить один `INSERT` без последующего `UPDATE`. Ограничение проявляется у обязательных, но переопределяемых дочерних строк.

Например, поездка может хранить локализации в отдельной таблице `localizations` с уникальным ключом `(rel_type, rel_id, language)`. Для каждой поездки по умолчанию должна существовать русская локализация, иногда с заданным в тесте текстом; дополнительно может понадобиться английская:

```php
$trip = TripFactory::new()
    ->withRussianTitle('Астрахань')
    ->withEnglishTitle('Astrakhan')
    ->create();
```

Фабрика должна до записи понимать окончательный набор локализаций и затем создать поездку и каждую локализацию одним `INSERT`, без корректирующих `UPDATE`. В стандартной фабрике русскую локализацию по умолчанию можно зарегистрировать через `has()`, но следующий `has()` не заменяет ее, а добавляет еще одну дочернюю фабрику. Попытка переопределить русский текст таким способом создаст две строки с одним языком и нарушит уникальный ключ. Если исправлять уже созданную default-строку в `afterCreating`, получится запрещенный контрактом `UPDATE`. Чтобы условно заменить ранее добавленную связь до создания, нужно самостоятельно управлять защищенной коллекцией `$has`, переопределять жизненный цикл фабрики или переносить эту логику в другой специальный слой.

Таким образом, Laravel позволяет соблюдать требование «один `INSERT` и ноль `UPDATE`» для заранее полностью описанного графа, но его публичный композиционный API не позволяет удобно задать обязательную дочернюю модель по умолчанию и затем заменить ее настройку. Собственные фабрики хранят окончательную конфигурацию связи как свое состояние и разрешают ее только в `create()`. Кроме этого:

- зависимости и допустимые способы их задания видны в сигнатурах методов;
- операции называются в терминах предметной области (`withTrip`, `withComment`, `admin`, `published`), а не выражаются произвольными массивами атрибутов;
- создание корректного графа инкапсулировано в фабрике верхнего уровня;
- фабрики находятся рядом со своим доменом и не требуют `HasFactory` в моделях;
- каждая модель полностью настраивается до сохранения, а callbacks после вставки используются только для новых дочерних строк и pivot-связей;
- API остается небольшим и адаптируется только к реально используемым сценариям проекта.

## Решение

Продолжать использовать собственные фабрики моделей вместо `Illuminate\Database\Eloquent\Factories\Factory`.

Считать основной единицей фабрики не отдельную модель, а пригодный для теста или сидера граф моделей. Фабрика верхнего уровня отвечает за разумные значения по умолчанию и за создание обязательных зависимостей согласно правилам предметной области. Вызывающий код переопределяет только важные для сценария детали.

Сохранить следующие соглашения:

- Фабрики располагаются в `app/Factory` или `app/Domain/*/Factory` рядом с соответствующей предметной областью.
- `new()` возвращает новую фабрику, методы настройки возвращают измененный клон, `make()` строит модель, `create()` сохраняет ее и завершает создание связанных данных.
- Произвольные атрибуты на манер Laravel (`create(['key' => 'value'])`) не используются.
- Для частых и значимых состояний используются именованные методы (`admin`, `inactive`, `hidden`, `obsolete`). Они составляют обнаруживаемый через API каталог типичных состояний модели и предпочтительнее низкоуровневой настройки отдельных атрибутов в вызывающем коде.
- Методы связей могут принимать готовую модель, идентификатор или фабрику зависимости. Вызов без аргумента означает создание разумной зависимости по умолчанию, если это однозначно для домена.
- Последний вызов настройки одной и той же связи должен побеждать: метод обязан очистить несовместимые варианты внутреннего состояния.
- Настройка фабрики не должна обращаться к базе данных. Создание зависимостей откладывается до `create()`.
- `make()` не выполняет запросов и всегда возвращает новую несохраненную корневую модель. Обязательные зависимости, которым нужен идентификатор из базы, разрешаются только в `create()`.
- До первого `save()` модель получает все обычные поля, статус, внешние ключи и значения, производные от зависимостей. Для каждой новой модели выполняется один `INSERT`, но не корректирующий `UPDATE`.
- Операции после сохранения родителя используются только там, где идентификатор родителя объективно необходим: для вставки дочерней модели или записи в pivot-таблицу. Они не должны исправлять атрибуты уже сохраненной модели.
- Если связь полезно строить как часть графа, ее метод принимает соответствующую фабрику. Это позволяет переопределить любые нужные поля зависимости до ее единственного `INSERT`.
- Владелец, локализованные заголовки, morph type/id, статусы, временные метки и имена файлов должны быть согласованы между всеми узлами автоматически созданного графа.
- Уникальные значения генерируются детерминированной последовательностью или достаточно большим collision-resistant значением, а не выбираются из маленького случайного диапазона.
- Случайные defaults вычисляются во время `make()` или `create()`, чтобы переиспользование настроенной фабрики не фиксировало случайное значение неожиданно.
- Фабрики используются только в тестах и сидерах (не в продакшне).

## Альтернативы

- **Стандартные фабрики Laravel**:
  - *Плюсы*: поддерживаются фреймворком; предоставляют `count`, `sequence`, states, callbacks, `for`, `has`, `hasAttached` и `recycle`; привычны разработчикам Laravel.
  - *Минусы*: значительная часть сценария выражается через массивы атрибутов, общие методы и magic methods; допустимые зависимости хуже видны из сигнатуры конкретной фабрики; бизнес-операции приходится дополнительно оборачивать именованными states и callbacks. `has()` добавляет дочерние фабрики, но не предоставляет публичной операции замены ранее заданной связи. Поэтому обязательную default-строку вроде русской локализации нельзя удобно переопределить, одновременно сохранив уникальность и требование «один `INSERT`, ноль `UPDATE`» для каждой модели. Решение требует либо отказаться от автоматического default, либо обновлять строку после создания, либо переопределять внутреннее устройство фабрики. Для текущего размера проекта миграция не дает достаточной выгоды.
- **Создавать модели непосредственно в тестах и сидерах**:
  - *Плюсы*: нет дополнительной абстракции, все значения видны в месте использования.
  - *Минусы*: дублируются обязательные поля и порядок создания зависимостей; тесты знают устройство базы вместо проверяемого сценария; изменение бизнес-правил требует массового редактирования фикстур.

## Последствия

**Позитивные:**

- Тесты и сидеры описывают значимые сценарии, а не порядок ручной вставки строк.
- Фабрика верхнего уровня инкапсулирует создание графа зависимостей и его значения по умолчанию.
- Для поддерживаемых связей зависимость можно заменить настроенной фабрикой и задать ее поля до единственного `INSERT`.
- Контракт запрещает корректирующие `UPDATE` после вставки модели ради применения состояния фабрики.
- Именованные, типизированные методы делают доступные состояния и связи обнаруживаемыми через IDE.
- Сценарии читаются в терминах предметной области: детали представления типичных состояний сосредоточены в фабрике и не дублируются в тестах и сидерах.
- Изменения правил построения тестовых данных локализованы в фабрике соответствующего домена.
- Неизменяемые методы настройки позволяют переиспользовать фабрики как шаблоны.

**Негативные:**

- Проект самостоятельно поддерживает инфраструктурный код, который частично дублирует возможности Laravel.
- Без общего проверяемого контракта поведение разных фабрик может расходиться.
- Автоматическое создание графа скрывает количество SQL-запросов и работу observers, поэтому фабрики не следует использовать в продакшн-коде обработки запросов.
- Разумные значения по умолчанию могут скрыть отсутствующую важную настройку теста. Значения, существенные для проверяемого поведения, необходимо задавать явно.
- Случайные данные усложняют воспроизведение ошибок, особенно для уникальных колонок; критичные значения должны быть детерминированными.
